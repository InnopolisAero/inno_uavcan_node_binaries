    # 6. Black-box testing guide

    TIP

    This is a black-box testing instructions, so it doesn't include neither unit nor integration tests.

    At the moment it is expected to manually execute all of the tests, but it should be iteratively automated soon.

    Each release binary should be thoroughly tested according to this pipeline to ensure the device's functionality, performance, and reliability.

    # Part 1. Simple check that the released binary version is correct

    # Test case 1-1. Normal

    WARNING

    There is a risk that the uploaded binary is wrong (wrong build flag, another node, previous version, etc).

    This test case is intented just to find it out.

    • erase chip memory (it's like a factory reset)
    • upload the latest firmware to the node
    • check the following properties using Yakut or Yukon:
      • node name
      • software version
      • hardware version
      • vssc
      • etc

    # Part 2. Compatibility/Migration Tests

    WARNING

    After uploading a new software, the registers values can be changed.

    Pay attention on safety critical registers.

    For example, mini v2 should not have def value != 1000, because it will trigger the motor to spin!

    # Test case 2-1. Migration drom DroneCAN

    • erase chip memory (it's like a factory reset)
    • upload the latest dronecan firmware
    • check the node is online
    • upload the latest firmware to the node
    • check the node is online and nothing is broken

    # Test case 2-2. Update version

    • erase chip memory (it's like a factory reset)
    • upload the previous cyphal version
    • check the node is online
    • upload the latest firmware to the node
    • check the node is online and nothing is broken

    # Part 3. Configuration Tests

    TIP

    There are 6 possible states:

    • Ublox can be either configured or not.
    • gps.config can be in default (0), force config (1) or done (2) states

    Provide tests for the most valuable states in the following sequence:

    # Test case 3-1. Factory state

    • erase chip memory (it's like a factory reset)
    • reset ublox using UART to the default, save the config and reboot it
    • upload the latest firmware to the node
    • wait for the configuration to succeed

    # Test case 3-2. Working state

    • reboot the node
    • it should automatically detect that the ublox is configured and skip the config process (the LED should blink purple, then blue and green)

    # Test case 3-3. Force config

    • reboot the node
    • it should automatically detect that the ublox is configured and skip the config process (the LED should blink purple, then blue and green)

    # Part 4. Functional Testing

    TIP

    Basic checks that the node interface matches the one declared in the documentation.

    # Test case 4-1. GNSS

    • Subscribe on all required topics:
      • zubax.gps.sats
      • zubax.gps.point
      • zubax.gps.yaw
      • zubax.gps.status
      • ds015.gps.*
    • Verify that the GNSS provides accurate position and timing information under different conditions

    # Test case 4-2. Magnetometer

    • Subscribe on all required topics
      • zubax.mag
    • Test the magnetometer by verifying its accuracy in determining direction and orientation

    # Test case 4-3. Barometer

    • Subscribe on all required topics
      • zubax.baro.press
      • zubax.baro.temp
    • Check the barometer's ability to measure changes in altitude and atmospheric pressure accurately

    # Part 5. Faults (self-diagnostic) Testing

    TIP

    Essential feature that helps to detect a hardware issue before shipping the device.

    # Test case 5-1. GNSS Hardware Issue

    • get the device with known GNSS hardware issue
    • erase chip memory (it's like a factory reset)
    • upload the latest firmware to the node
    • it should be complitely clear for a user that the device has a hardware issue

    # Test case 5-2. Magnetometer Hardware Issue

    • get the device with known Magnetometer hardware issue
    • erase chip memory (it's like a factory reset)
    • upload the latest firmware to the node
    • it should be complitely clear for a user that the device has a hardware issue
